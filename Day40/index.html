How this was built ▶

System Prompt Design

The underlying Claude call is scoped narrowly to one job: scoring a resume against a job description for ATS (Applicant Tracking System) compatibility and keyword match, aimed at students/early-career candidates. It's instructed to:

  • Always return strict JSON matching a fixed schema (overall score, verdict, 4 subscores, prioritized fixes, keyword match list, before/after rewrites, section-by-section summary) so the UI can render it deterministically without fragile text-parsing.
  • Refuse gracefully (via an error field, never a broken JSON) when the input isn't actually a resume, is empty, or the job description is missing — the UI checks for this field before rendering.
  • Stay encouraging but honest in tone — polished career-advisor voice, not harsh, since the target user is early-career and may be reviewing their first resume.
  • Never fabricate resume content that wasn't provided; rewrites must only rephrase existing bullets, not invent new experience.

UI Choices

  • Score dial instead of a plain number — gives an instant pass/borderline/fail read, the way a real ATS score-check tool would.
  • Subscore bars (keyword match, formatting, impact language, structure) break the single score into actionable dimensions instead of one opaque number.
  • Prioritized fix list with High/Medium/Low tags mirrors how a career coach would triage feedback — fix the resume-killers first.
  • Keyword chips (present vs. missing) make the JD-matching concrete and scannable rather than buried in prose.
  • Before/after rewrite cards give copy-pasteable improvements, since the #1 outcome for this user is "leave with a better resume," not just a diagnosis.

Extending This

  • Real file parsing: swap the plain-text file reader for a PDF/DOCX extraction library (e.g. pdf.js, mammoth.js) loaded from a CDN to handle real uploads instead of relying on paste-fallback.
  • Multi-step flow: add a second call that takes the accepted rewrites and regenerates a full clean resume draft, or a "re-check" button that re-scores after edits.
  • Memory: persist past reports (localStorage or a backend) so returning users can track score improvement over multiple drafts.
  • Tools: add a web_search tool to the API call so the model can pull in live examples of strong bullet phrasing for the candidate's specific target role/industry.
